# Workflow и правила правок

## Главная цель

Сайт должен быть приятным не только для пользователя, но и для тебя как для хозяйки репозитория.

Правка не должна вызывать страх, что сейчас что-то сломается.

## Главное правило

Любое изменение должно попадать только в одну из трёх зон:

1. Контент
2. Стиль
3. Логика

Если одна мелкая правка требует лезть в пять мест — архитектура плохая.

## Где что менять

### Меняем текст первого экрана
Идём в `data/site.json`

### Меняем правило анализа
Идём в `data/detector-rules.json`

### Меняем цвета, радиусы, размеры
Идём в `styles/tokens.css`

### Меняем раскладку экрана
Идём в `styles/layout.css`

### Меняем вид карточек, pills, topbar
Идём в `styles/components.css`

### Меняем поведение детектора
Идём в `scripts/detector.js`

## Как делать коммиты

Не писать:
- fix
- update
- changes

Писать:
- `docs: add content model`
- `feat: add detector rules json`
- `style: refine mobile hero spacing`
- `fix: prevent detector overflow on small screens`
- `refactor: move hero copy to site json`

## Как делать ветки

### main
Только стабильное

### dev
Рабочая сборка

### feature-ветки
Одна ветка = одна задача

Примеры:
- `feature/home-v1`
- `feature/detector-v1`
- `feature/dossiers-v1`

## Что нельзя делать

- не смешивать логику и тексты в одном JS-файле
- не копировать одинаковые стили по разным страницам
- не держать всё в одном `index.html`
- не рисовать новую кнопку вручную на каждом экране
- не хардкодить часто меняющиеся тексты

## Проверка перед коммитом

Каждую правку проверять минимум на:
- 390px
- 430px
- desktop

И спрашивать:
- не появился ли скролл там, где его не должно быть
- не поехал ли текст
- не стал ли экран шумным
- не стало ли два одинаково важных CTA

## Порядок разработки

1. Сначала документы и логика
2. Потом данные
3. Потом визуальная система
4. Потом экран
5. Потом интерактив
6. Потом контент слоями

## Вывод

Opticanew нельзя собирать как быстрый одноразовый landing.
Его нужно строить как спокойную, управляемую систему.
